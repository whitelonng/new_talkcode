一、Agent 定位
browser-control 的职责是：

导航页面
检查页面状态
读取 DOM / 页面结构
执行用户式交互
调试 console / network
执行脚本辅助诊断
验证页面操作结果
复现 UI 问题并给出证据
它的推荐工作方式是：

先理解目标
先导航/等待稳定
先读后动
每一步都做验证
输出结论、证据、阻塞点

三、能力边界标准
当前浏览器控制不是所有页面都“完全可控”。系统里已经明确了 5 种状态/模式。

页面模式
none
fileControlled
localhostControlled
externalEmbedded
externalNativeControlled
能力集
每个 session 会带能力集：

navigation
domRead
domWrite
scriptEval
consoleRead
networkObserve
screenshot
keyboardInput
mouseInput
externalControl
能力状态不是简单二值，而是：

available
partial
unavailable
你要用的判断标准
完全可控页
通常是：

fileControlled
localhostControlled
要求至少具备：

domRead = available
domWrite = available
scriptEval = available
这类页可以做：

快照
元素发现
点/填/查/等待
大部分验证闭环
半可控页
常见是：

externalEmbedded
externalNativeControlled
可能仅具备：

navigation
keyboardInput
mouseInput
部分 external control
这类页可能：

能打开
能切页
能模拟部分交互
但不能稳定读 DOM / snapshot / query
不可控页
表现通常是：

BRIDGE_NOT_READY
CAPABILITY_UNAVAILABLE
COMMAND_TIMED_OUT
COMMAND_ALREADY_PENDING
这类页不能假装继续做“读到了/验证到了”。

四、Browser Control 要测试什么
你要测试的不是“工具存在没有”，而是三层：

功能存在
功能在可控页能稳定工作
异常情况下能恢复，不锁死

五、详细测试流程标准
下面给你一份可直接执行的Browser Control 测试标准流程。

A. 测试目标
验证 Browser Control Agent 在当前版本下是否满足：

能打开页面
能建立 bridge
能持续读取页面
能执行用户交互
能验证动作结果
能读取 console / network
出现 timeout / session reset / stale pending 时能恢复

目标：

用途：

D. 详细测试项
D1. 基础连通性测试（缩减版）
目标
验证浏览器能否打开并形成 session，同时把重点放在“打开后是否真正进入可控状态”。

步骤
调用 browserNavigate
对 fileControlled / localhostControlled 继续调用 browserGetPageState
记录：
mode
status
url
title
capabilities
sourceType
currentFilePath（仅 fileControlled）
通过标准
页面能成功打开（该项已在日志中验证通过，本轮只需确认未回退）
fileControlled / localhostControlled 下 browserGetPageState 能返回结构化状态
session 非空
mode 合理
status 最终可进入 ready（对于 externalNativeControlled，ready 代表原生 session 就绪，不代表 DOM bridge 已可读）
对于 `browserNavigate(file://...)`：
- 必须确认其最终进入 `sourceType='file'`
- 不得停留在旧的 `sourceType='url'` 语义
- `currentFilePath` 应与目标本地文件一致
- 仅“页面成功打开”不能判定 fileControlled 主链路通过，必须继续确认 bridge ready 与 page state 可读

失败判定
页面能打开，但打开后不能读状态
一直卡在 loading
直接进入 error/failed
BRIDGE_NOT_READY
Browser bridge command timed out.
Another browser bridge command is still pending.

补充说明
- “仅能否导航成功” 已在日志中通过，不再单独作为主测试项。
- fileControlled / localhostControlled：必须验证 bridge ready 与 runtime ready 两阶段都能完成。
- 若 fileControlled 场景中 `browserGetPageState` / `browserSnapshot` / `browserWaitFor` 无法执行，应记录为主链路失败，而不是仅记为临时波动。
- externalNativeControlled：允许不具备 DOM 读取能力，但必须验证 session 可启动、可刷新状态、可关闭、可继续导航。
- `file://` 本地文件页当前仍是必须重点回归的高风险链路，不得默认视为已稳定通过。
D2. 读取链路测试
目标
验证“读页面主链路”是否稳定。

工具
browserGetPageState
browserSnapshot
browserListInteractiveElements
browserGetElementInfo
browserQueryElements
browserGetDomTree
步骤
打开测试页
连续执行：
browserGetPageState
browserSnapshot
browserListInteractiveElements
browserQueryElements
browserGetDomTree
对一个目标元素执行：
browserGetElementInfo
通过标准
连续读取不出现 pending 卡死
snapshot 能返回有效页面信息
interactive elements 非空且合理
dom tree 能返回结构
element info 能返回 selector 对应信息
失败判定
COMMAND_ALREADY_PENDING
COMMAND_TIMED_OUT
CAPABILITY_UNAVAILABLE
前一个命令失败导致后续全部阻塞

适用范围
- 本组测试仅对 `fileControlled` 与 `localhostControlled` 做强制要求
- `externalNativeControlled` 仅需验证能力边界返回正确，不要求 DOM 读取成功
D3. 单元素交互测试

验证点击/输入/聚焦/键盘动作是否可靠。

工具

browserClickByText
browserType
browserFocus
browserBlur
browserHover

highlight

必须在每一步后执行验证：
browserGetPageState
browserSnapshot
browserWaitForText
或 browserWaitForElementState
通过标准
动作成功执行
页面有可验证变化
无异常残留 pending
下一个命令还能继续
失败判定
动作发出但无结果
动作后无法确认页面变化

输入焦点丢失且无法恢复

工具

browserGetPageState
browserSnapshot

定位表单元素

或等待文本出现

必须能回答以下问题：

是否出现错误提示文案
是否发生跳转

D5. 调试能力测试（缩减版）
目标
验证 console / network 不只是“接口可调用”，而是能形成有效观测、可清理、可定位。

工具
browserGetConsole
browserGetConsoleErrors
browserClearConsole
browserGetNetworkLogs
browserFindNetworkRequest
browserGetRequestDetail
browserClearNetworkLogs
Console 测试步骤
执行页面操作

Network 测试步骤
触发一个明确请求
读取 network logs
用 urlIncludes / method / status 查找请求
读取 request detail
清空 network logs
再次验证已清空
通过标准
不是只验证接口能调用，而是要拿到与页面行为一致的有效日志
能查到目标请求
detail 数据结构完整
clear 后状态正确
失败判定
一直返回空，且与页面实际行为不符
detail 取不到
clear 不生效
读调试信息会干扰后续命令

补充说明
- `browserGetConsoleErrors`、`browserGetNetworkLogs` 不能仅以“接口可调用”视为通过。
- 对于当前 fileControlled 主链路，必须验证：
  - 是否能拿到与页面行为一致的有效日志
  - 是否能 clear
  - 是否能定位具体请求 / 错误
- 若 bridge 未 ready 导致 debug 链路无法进入，应按主链路失败记录，而不是跳过。
D6. 等待类工具测试

工具

browserWaitForNavigation
browserWaitForText
browserWaitForElementState
步骤
触发会延迟变化的动作

selector 出现

通过标准
等待工具能正确结束
超时时能明确报错
不会造成 pending 永久残留
失败判定
等待假成功
等待超时后后续不能继续
状态变化发生了但工具没识别
D7. 恢复能力测试
这是当前最重要的一组。

目标
验证出现异常后系统不会锁死。

必测场景 1：命令超时后恢复
步骤：
- 人为构造超时命令
- 记录 pending 是否被释放
- 再发起下一条命令
- 验证下一条命令可继续成功或明确失败，不得残留锁死状态

必测场景 2：session reset 后恢复
步骤：
- 在 bridge ready 后切换 session / 刷新 iframe / 重新导航
- 验证旧 pending 命令被拒绝
- 验证 stale result 不会污染新 session
- 验证新 session 可以重新完成 ready 握手

必测场景 3：runtime ready 事件丢失后的恢复
步骤：
- 验证 fileControlled / localhostControlled 的 bridge runtime 会重复发送 ready 握手
- 验证父端能在稍晚时机收到 ready 并恢复到可执行状态
- 验证不会永久停留在 initializing / loading

必测场景 4：native session 状态同步
步骤：
- 启动 externalNativeControlled 会话
- 触发原生导航 / 刷新状态 / 关闭会话
- 验证 native state changed 事件会同步更新地址栏、title、status、capabilitySet
- 验证只同步当前 active session，旧 session 事件不会污染当前状态

通过标准
- 超时、session reset、stale pending、迟到的 ready 事件、native state changed 都不会造成系统锁死
- 下一条命令仍可继续执行
- store / UI / session meta 状态一致

失败判定
- pending 无法释放
- 新 session 无法接管
- ready 事件补发后仍无法恢复
- native session 已变化但 UI / 状态未更新
- 旧 session 事件覆盖当前 session 状态

触发一个会超时的命令
确认返回 COMMAND_TIMED_OUT
立刻执行新命令
检查是否能继续成功
通过标准：

超时后 pending 被释放
新命令可继续执行
必测场景 2：session reset 后恢复
步骤：

让当前命令处于 pending
触发 session rotate / reset
检查旧命令是否收到明确拒绝
再发新命令
通过标准：

旧命令被标记 SESSION_RESET
pending 被清空
新命令能继续
必测场景 3：旧结果晚到不污染当前状态
步骤：

发命令 A
session reset
发命令 B
再让 A 的旧 result 晚到
检查当前状态是否仍属于 B
通过标准：

A 的旧结果被忽略
当前 session/result 不被污染
必测场景 4：runtime ready 与 bridge ready 一致性
步骤：

页面打开后立刻发读取命令
预期在 runtime 未 ready 时被拒绝
收到 ready 事件后重新发命令
应成功
通过标准：

不会误把假 ready 当真 ready
真正 ready 后可以继续执行

必须按顺序测，不能跳。

第 1 轮：可控页基线测试
先测：

fileControlled
localhostControlled
顺序：

导航

补充要求
- `fileControlled` 必须优先覆盖 `browserNavigate(file://...)` 场景，确认页面打开后不只是“可见”，还要验证是否真正进入 bridge ready。

第 2 轮：外部页边界测试
再测：

externalEmbedded
externalNativeControlled
重点不是追求全功能成功，而是验证：

哪些能力 partial
哪些工具应明确拒绝
拒绝是否准确、可恢复
第 3 轮：回归测试
每次修 bridge 后，至少回归：

browserGetPageState
browserSnapshot
browserListInteractiveElements
timeout 恢复
session reset 恢复
stale result 忽略

补充回归项（本轮 file:// 高风险链路）
- `browserNavigate(file://...)` → `sourceType='file'`
- `currentFilePath` 正确写入
- bridge ready / runtime ready 真正完成，而不是仅页面 load
- file preview 页面可以成功执行 `browserGetPageState`
- file preview 页面可以成功执行 `browserSnapshot`
- timeout 后 pending 可释放，不阻塞下一条命令
- 重新导航后若页面重新打开，需继续验证能否恢复到 DOM 可读 / 可交互状态，而不只是恢复可见页面
