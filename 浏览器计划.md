# 内置浏览器控制能力实施计划

## 1. 背景

当前 TalkCody 已具备内置浏览器、browser bridge、浏览器专用 agent 以及一整套浏览器工具，但真实体验仍存在明显断层：

- 启动后虽然可以在应用内打开页面，但很多页面并不能稳定“读到、点到、填到、验证到”
- 用户容易误以为只要存在 `browserSnapshot`、`browserClickByText`、`browserFillForm` 等工具，就代表页面一定可控
- 当前 bridge 采用严格单命令 pending 模型，一旦前序命令超时或卡住，后续命令会整体阻塞
- 外部页面与本地可控页面的能力边界没有被产品化表达清楚，导致执行结果经常看起来像“工具失灵”

因此，这个计划的目标不是再新增一套浏览器系统，而是把现有 **内置浏览器 + browser bridge + browser-control agent** 打磨成一条可稳定工作的主路径。

---

## 2. 当前代码事实

### 2.1 已有专门的浏览器控制 agent

当前项目里已经存在专门的浏览器控制 agent：

- 文件：`src/services/agents/browser-control-agent.ts`
- `id: 'browser-control'`
- `name: '浏览器控制'`
- `role: 'write'`

它已挂载完整浏览器工具：

- 导航：`browserNavigate`、`browserWaitForNavigation`
- 页面读取：`browserGetPageState`、`browserSnapshot`
- 元素发现：`browserListInteractiveElements`、`browserQueryElements`、`browserGetElementInfo`、`browserGetDomTree`
- 交互：`browserClick`、`browserClickByText`、`browserType`、`browserFillForm`
- 调试：`browserGetConsole`、`browserGetConsoleErrors`、`browserGetNetworkLogs`、`browserFindNetworkRequest`、`browserGetRequestDetail`
- 补充动作：`browserFocus`、`browserBlur`、`browserHover`、`browserSelectOption`、`browserCheck`、`browserUncheck`

结论：

**当前不是没有浏览器控制主 agent，而是已经有 `browser-control`。**

---

### 2.2 已有内置浏览器桥接体系

核心文件：

- `src/components/browser/browser-panel.tsx`
- `src/services/browser-bridge-service.ts`
- `src/stores/browser-store.ts`
- `src/types/browser-control.ts`

当前体系已支持：

- 在应用内打开 URL 或文件页面
- 建立 bridge session
- 派发 browser command
- 回收 browser result
- 收集 console / network 日志
- 区分不同运行模式和 capability set

结论：

**“启动后不需要打开外部浏览器”与当前架构完全一致。**

## 3. 问题定义

你列出的“当前还做不到的能力”，本质不是一个问题，而是三类问题叠加：

### 3.1 页面能力边界不一致
当前存在多种运行模式：

- `fileControlled`
- `localhostControlled`
- `externalEmbedded`
- `externalNativeControlled`

不是每种模式都具备完整：

- `domRead`
- `domWrite`
- `scriptEval`
- `consoleRead`
- `networkObserve`

因此：

- 某些页面只能导航，不能稳定读 DOM
- 某些页面能输入鼠标键盘，但不能读取正文和按钮文案
- 某些页面只能部分调试，不能完整验证行为结果

---

### 3.2 bridge 稳定性不足
从现有实现与错误信息看，bridge 还存在以下显著问题：

#### A. 命令超时
证据：

- `Browser bridge command timed out.`

含义：

- command 发出后没有收到 result
- bridge ready 不代表页面端真正可执行
- 页面切换 / session rotate 后可能出现结果丢失

#### B. ready 与 runtime ready 的门槛耦合过严
来自 `logs/浏览器日志.md` 的现象：

- 页面能打开
- `browserGetPageState` / `browserSnapshot` / `browserListInteractiveElements` / `browserWaitFor*` 大量报：
  - `Browser bridge is not ready. Current status: loading`
  - `Browser bridge is not ready. Current status: error`

结合 `src/services/browser-bridge-service.ts` 可确认，旧链路的关键矛盾是：

- 页面已经发出 `ready` 事件，store 已进入 `bridgeStatus = 'ready'`
- 但 `executeCommand()` 仍额外依赖内部 `runtimeState === 'ready'`
- 命令发送阶段又会通过 `markRuntimeLoading(...)` 把 runtime 状态打回 `loading`

结果是：

- 首个读取类命令会被错误地拦在 service preflight 阶段
- 日志表现为“可以导航，但读不到页面”
- `localhostControlled` / `externalEmbedded` 更容易长期停在 `loading`

#### C. 当前页能力不足时命令直接不可用
证据：

- `Browser bridge command "snapshot" is not available for current page.`
- `Browser bridge command "clickByText" is not available for current page.`

含义：

- 问题不是工具缺失
- 而是当前页 session 没有拿到 DOM 控制能力
- 命令会在 service 层被提前拒绝

#### D. 串行 pending 模型导致整段流程容易锁死
证据：

- `Another browser bridge command is still pending.`

含义：

- 当前同一时刻只允许一个命令在路上
- 任何命令卡住，后续读、点、填、查都会被一并阻塞
- 缺少更强的超时恢复与陈旧命令清理机制

---

### 3.3 产品层没有把“可控/半可控/不可控”讲清楚
虽然 prompt 层已经提示：

- 优先走 `fileControlled` / `localhostControlled`
- DOM read 不可用时不要假装看得见页面
- 应该先读后动

但这些规则目前主要存在于 prompt 和内部实现，用户感知不足。

结果就是：

- 用户以为外部页面也能像本地预览页一样稳定自动化
- 拿到 0 条 console/network 结果时会误判为页面没问题
- 工具失败时难以区分是“页面不可控”还是“bridge 异常”

---

## 4. 范围定义

## 4.1 本计划要解决的范围

- 内置浏览器作为浏览器任务默认路径
- browser bridge 读取链路稳定化
- pending / timeout / session reset 的恢复机制
- 提交后验证闭环
- 可控性状态透明化

## 4.2 本计划暂不解决的范围

- 对所有互联网外部页面提供完整 DOM 自动化保证
- 绕过页面安全策略获取不可访问 DOM
- 做成通用 RPA 产品
- 对复杂验证码、人机验证提供自动破解能力

---

## 5. 总体目标

### 目标 G1：内置浏览器成为默认主路径
- 启动后不需要再打开外部系统浏览器
- 浏览器任务默认由应用内浏览器承接
- 用户能直接看到当前页面的可控状态

### 目标 G2：读页面主链路稳定可用
在可控页面上，以下动作需要连续稳定完成：

1. `browserGetPageState`
2. `browserSnapshot`
3. `browserListInteractiveElements`

### 目标 G3：动作后验证闭环可用
在可控页面上，能够完成：

- 填表
- 点击提交
- 等待状态变化
- 读取错误/成功证据
- 辅助查看 network / console

### 目标 G4：bridge 不再因单命令卡死拖垮整段流程
- timeout 后能恢复
- session reset 时能清理旧命令
- 不因残留 pending 长时间阻塞所有后续动作

## 6. 验收标准

### 6.1 文档验收标准
- 文档明确区分当前能力、问题根因、阶段路线、验收标准
- 文档结论与当前代码一致，不再把 `coding` 误写成唯一主浏览器 agent
- 文档可以直接作为研发执行单继续拆任务

### 6.2 实现验收标准（第一阶段）
在可控页面或测试环境中：

- 连续触发读取类命令时，不因上一次残留 pending 长期锁死
- timeout 后 store 与 service 状态能恢复到可继续执行
- session reset 后旧命令被明确拒绝，并释放 pending
- ready 后首个读取命令不会再被额外 runtime ready 门槛错误拦截
- 相关测试覆盖超时、session reset、命令恢复场景

### 6.3 产品体验验收标准（后续阶段）
- 用户能清楚知道当前页是“可控 / 部分可控 / 不可控”
- 当 DOM read 不可用时，系统不会继续假装安全操作
- 登录类流程能输出结构化结果：成功 / 失败 / 二次验证 / 不可控阻塞

## 8. 第一优先级落地项

### P0-1：修复 pending command 卡死后的恢复能力
原因：

- 当前这是最高优先级问题之一
- 它会同时破坏读取、点击、填表、调试所有路径
- `logs/浏览器日志.md` 已明确三组页面在失败后均未恢复成功：
  - `fileControlled` 保持 `error`
  - `localhostControlled` / `external` 保持 `loading`
- 如果 pending / timeout / stale result 生命周期不稳定，会持续放大日志噪音，干扰对真实页面注入链路的判断

计划动作：

- 审查 `browser-bridge-service.ts` 的 pending 生命周期
- 增强 timeout 后的状态恢复
- 增强 session reset 后的命令清理
- 必要时补充“清理陈旧 pending”的保护逻辑
- 补充测试覆盖以下场景：
  - 超时后再次执行新命令
  - session reset 后再次执行新命令
  - 旧命令结果晚到时不污染当前状态

验收标准：

- pending 不会因为单次异常长期残留
- 新命令可以在恢复后继续执行
- 旧结果不会污染新 session / 新命令状态
- 真实页面排查时，可以明确区分“bridge 未注入”和“链路被旧 pending 卡死”

状态：

- 仍为当前最高优先级之一
- 应与注入链路排查并行推进，避免恢复能力缺口持续放大日志噪音

- 已明确 `fileControlled` 进入 `error` 的根因并形成修复项
- 已明确 `localhostControlled` / `external` 长期 `loading` 的根因并形成修复项
- pending / session reset / stale result 的恢复测试补齐，且不会再干扰真实页面根因判断
- 已明确 `console/network` 返回 `0 条` 的真实含义，并能区分“页面无事件”与“观测链路未建立”

## 11. Agent 分工

### `browser-control`
- 浏览器主控制 agent
- 负责导航、读取、交互、验证、调试

### `coding`
- 负责根据浏览器发现修代码
- 修完后回归验证

### `general`
- 适合轻量读取与问答
- 不应承担关键自动化流程

### `orchestrator`
- 负责任务编排
- 决定何时调用 `browser-control`、何时切到 `coding`

## 12. 最终结论

当前项目的问题不是“还没有浏览器控制能力”，而是：

1. **已有能力的边界没有被清晰表达**
2. **bridge 稳定性还不足以支撑可靠自动化闭环**
3. **`ready -> runtime ready` 的旧 preflight 误拦截问题已经完成代码与测试收口**
4. **最新日志表明，当前最高优先级已转为恢复链路与真实页面注入链路联合定位：file 直接 `error`，localhost / external 长期 `loading`**
5. **console / network 虽有接口入口，但 `0 条` 结果说明调试观测闭环仍未建立，暂不能视为真实可用**

因此，正确方向是：

- 默认走内置浏览器
- 用 `browser-control` 作为浏览器主控 agent
- 优先补强 pending / timeout / session reset / stale result 恢复能力
- 并行定位 file / localhost / external 三类页面桥接注入分化的真实根因
- 单独验证 console / network 观测闭环，避免把“接口可调用”误判为“调试可用”
- 在根因明确后，再逐步补齐读取、等待、交互、调试与提交验证闭环

一句话总结：

**现在最该做的是先把恢复链路做稳，再把真实页面上的 bridge 注入、状态上报和观测闭环查清。**
